# Подключение Supabase

Backend для совместного редактирования вчетвером + общие скрины + аудит.
Всё инженерное уже готово в репо: схема, роли, миграция данных, хранилище.
От тебя — один разовый шаг (создать проект). Дальше всё запускаю я.

## Что нужно от тебя (~5 минут)

1. **Создать проект.** Зайти на [supabase.com](https://supabase.com) → *New project*.
   Бесплатного тарифа хватает. Region — ближайший (например Frankfurt).
   Придумать и сохранить пароль БД.

2. **Значения.** URL и `anon`-ключ — из *Project Settings → API*.
   Строка подключения с паролем — из *Project Settings → Database →
   Connection string (URI)*.

3. **В файл `.env.local`** (в корне репо), пароль — только сюда, не в чат:
   ```
   NEXT_PUBLIC_SUPABASE_URL=https://xxxxx.supabase.co
   NEXT_PUBLIC_SUPABASE_ANON_KEY=sb_publishable_...
   DATABASE_URL=postgresql://postgres:ПАРОЛЬ@db.xxxxx.supabase.co:5432/postgres
   ```
   `.env.local` в `.gitignore` — в git не попадёт.

## Что дальше делаю я

4. **Применяю схему** (создаёт таблицы, роли, политики, bucket `screens`):
   ```bash
   node --env-file=.env.local supabase/migrate.mjs
   ```

5. **Заливаю данные** (169 каналов + спринт + интеграции):
   ```bash
   node --env-file=.env.local supabase/seed.mjs
   ```

6. **Включаю вход и запись в приложении.** Логин по email-ссылке (magic-link),
   редактор результатов и доска начинают писать в БД вместо localStorage,
   скрины грузятся в хранилище. Тестируем вместе.

## Роли (раздаются в таблице `profiles`)

| Роль | Доступ |
|---|---|
| `admin` | всё + управление людьми (первый зарегистрированный — автоматически admin) |
| `producer` | вести спринт и результаты |
| `researcher` | заводить и править каналы в бэклоге |
| `viewer` | только чтение (заказчик/гость) |

Чтобы пригласить троих: они заходят на сайт, логинятся по своей почте →
ты в таблице `profiles` ставишь им нужную роль.

## Деплой (потом)

Фронт — на Vercel (импорт репо, те же env-переменные), данные остаются в Supabase.
Команда заходит по ссылке вида `kaiten-influence.vercel.app`.
