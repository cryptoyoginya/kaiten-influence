## Как происходит сейчас

### Скоринг креативов

| Что | Как | Чем |
|---|---|---|
| Генерация сета | Агент (Claude Code) пишет 5 вариантов по осям разнообразия | правило [creative-diversity.md](creative-diversity.md), голос-профиль канала |
| Гейты до скоринга | Diversity (скелет/шафл/открытия/CTA) → Claims-разметка FACT/INF/RISK по строкам KB → Integrity | KB content_zavod (product-facts, редполитика, ICP) |
| Оценка | Агент по рубрике: 6 линз × веса → composite, зоны 80/60 | рубрика калибрована на датасете реальных офферов |
| Voice-match | 5 чек-пунктов × 20 по голос-профилю, гейт ≥70 | корпус реальных постов канала (`content/channels/<slug>/real-posts.md`) |
| Панель ЦА | 7 синтетических персон сегмента: мысли от первого лица + достаёт/говорит + прогноз действия | `content/segments/<seg>/panel.md`, портреты из ICP и данных канала |
| Adversarial | 3 скептика на HIGH-находки, подтверждение ≥2/3 | отклонённые претензии фиксируются в отчёте |
| Выход | HTML-отчёт (10 секций: линзы, voice, панель, бенчмарк, риски, правки, A/B-прогноз, интейк) | генератор → `public/scoring/*.html`, на проде `/scoring/...` |
| Предсказание | ab-plan.json: ранжирование, гипотезы, целевые персоны — до публикации | `content/channels/<slug>/sets/<set>/ab-plan.json` |

**Ключевой принцип:** это LLM-судья с рубрикой, не ML-модель. «Обучение» =
калибровка рубрики, портретов и голос-профилей на размеченных данных,
с петлёй «предсказание → факт → правка модели» после каждого размещения.

### Аналитика конкурентов

1. Агент снимает офферы страниц конкурентов (главная + тарифы) — **дословные
   цитаты**, с URL и датой снятия.
2. Каждая цитата размечается FACT/INF/RISK, оффер скорится **той же рубрикой**,
   что наши креативы → честный бенчмарк в отчёте.
3. Файлы посегментные: `content/segments/<seg>/competitors.json` — редакция
   сравнивается с Notion-заменами, IT — с трекерами.
4. Валидатор `node scripts/check-offers.mjs`: пересчёт композитов, свежесть
   (порог 30 дней), схема, покрытие каналов.

Текущая база: **12 брендов** (editorial: Яндекс Вики, Teamly, Yonote, Minerva ·
it-dev: Shtab, YouGile, EvaTeam, Яндекс Трекер, Planfix, Аспро.Cloud,
Мегаплан, Битрикс24). Ни один оффер рынка не в зелёной зоне (максимум 77.5).

### Честные ограничения текущей версии

- Снятие офферов — ручное (агент + WebFetch): некоторые сайты закрывают
  соединение (WEEEK — 3 попытки), динамические страницы отдаются не полностью.
- История изменений чужих офферов не накапливается автоматически.
- Факты размещений (CTR, пересылки) ещё не поступали — петля калибровки
  собрана, но не прогонялась на реальных цифрах.
- TGStat-статистика каналов не подключена: прогноз охвата со слов продавца.

## Что планируется: парсер офферов

Автоматизация шага «снятие и слежение». Суждение (разметка и скоринг)
остаётся за агентом — это его работа.

### Спецификация `scripts/parse-offers.mjs`

```
node scripts/parse-offers.mjs                # все сегменты
node scripts/parse-offers.mjs editorial      # один сегмент
```

1. **Вход:** список URL из `competitors.json` каждого сегмента (поле `source`).
2. **Снятие:** headless Chrome (уже используется для рендера визуалов) —
   обходит анти-бот и дорисовывает динамические страницы, которые не отдаются
   обычному fetch (кейс WEEEK).
3. **Снапшоты:** `content/segments/<seg>/snapshots/<brand>/<YYYY-MM-DD>.md`
   (извлечённый текст страницы; HTML не храним — шумно и тяжело).
4. **Дифф:** сравнение с предыдущим снапшотом. Изменился текст в зоне цитат →
   пометка в `competitors.json`: `stale: true` + отчёт «что изменилось».
5. **History:** при подтверждении изменения старая цитата уезжает в
   `history[]` с датами — динамика чужого позиционирования накапливается сама.
6. **Запуск:** вручную перед закупкой + по расписанию раз в 30 дней
   (cron/Action). Выход — дифф-отчёт: «Teamly сменил слоган, пересмотреть
   скоринг» вместо молчаливого протухания.

### Что парсер НЕ делает

Не скорит и не размечает: суждение (FACT/RISK, линзы, уроки) остаётся
за агентом по методике [COMPETITORS.md](COMPETITORS.md). Парсер гарантирует,
что суждение выносится по свежему материалу.

### Очередь автоматизации после парсера

| Приоритет | Что | Зачем |
|---|---|---|
| P1 | Парсер офферов (спека выше) | база конкурентов перестаёт протухать |
| P2 | TGStat API → channel.md | прогноз охвата из данных, не со слов продавца |
| P3 | Автосбор факта по UTM после публикации → ab-plan.facts | петля калибровки замыкается без ручного ввода |
| P4 | Снапшот лендинга-цели при прогоне | Claims-линза ловит разрыв «пост обещает — лендинг молчит» автоматически |
